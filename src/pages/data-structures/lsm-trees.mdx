# Log-Structured Merge-Trees (LSM Trees)

LSM Trees are the data structure underlying many modern, write-optimized NoSQL databases like Cassandra, RocksDB, and LevelDB.

## The Problem with B-Trees

Traditional relational databases use B-Trees. When you write to a B-Tree, the database has to find the exact location on disk to place the data. This results in **random disk I/O**, which is very slow, especially on spinning hard drives.

## How LSM Trees Work

LSM Trees solve this by turning random writes into **sequential writes**.

1.  **MemTable (In-Memory):** All incoming writes are first appended to an in-memory balanced tree (like a Red-Black tree). Because it's in memory, it's extremely fast.
2.  **Commit Log (On-Disk):** To prevent data loss if the server crashes, every write is also sequentially appended to a log file on disk.
3.  **SSTable (On-Disk):** When the MemTable gets too large, it is flushed to disk as an Immutable Sorted String Table (SSTable). Because the MemTable was already sorted, flushing it is a fast sequential write.
4.  **Compaction:** Over time, you accumulate many SSTables. A background process called "compaction" merges these tables together, removing deleted data (tombstones) and keeping the data sorted.

### Reading Data
To read a key, the database checks:
1. The MemTable (fastest).
2. The most recent SSTable.
3. Older SSTables.
*(Bloom filters are heavily used here to quickly skip SSTables that definitely don't contain the key).*

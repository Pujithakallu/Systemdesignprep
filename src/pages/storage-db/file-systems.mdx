# File Systems Design

A file system dictates how data is stored and retrieved on a storage medium (like a hard drive or SSD). 

## Core Components

1.  **Data Blocks:** The actual raw data of the file, split into fixed-size blocks (e.g., 4KB).
2.  **Inodes (Index Nodes):** Metadata about the file. An inode contains:
    - File size, permissions, owner, timestamps.
    - **Pointers** to the data blocks where the actual file content lives.
    - *Note: The file name is NOT stored in the inode.*
3.  **Directories:** A directory is just a special type of file. Its data blocks contain a list of mappings: `(File Name -> Inode Number)`.
4.  **Superblock:** Contains metadata about the entire file system (total blocks, free blocks, root inode location).

## How a File is Read

When you read `/home/user/doc.txt`:
1.  Read the Superblock to find the Root Directory (`/`) inode.
2.  Read the `/` directory data blocks to find the inode number for `home`.
3.  Read the `home` directory data blocks to find the inode number for `user`.
4.  Read the `user` directory data blocks to find the inode number for `doc.txt`.
5.  Read the `doc.txt` inode to find the pointers to its data blocks.
6.  Read the data blocks to get the text.

## Distributed File Systems (e.g., HDFS, GFS)

For massive scale, a single disk isn't enough. Distributed File Systems split large files into massive chunks (e.g., 64MB or 128MB) and distribute them across hundreds of commodity servers.
- **NameNode (Master):** Stores the metadata (which chunks belong to which file, and which servers hold those chunks).
- **DataNodes (Workers):** Store the actual data chunks.

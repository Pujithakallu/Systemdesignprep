# Clarifying Inputs

Before you design anything, you must ask the interviewer questions to define the constraints of the system. This is called "Back-of-the-envelope estimation."

## 1. Traffic & Usage
- **DAU (Daily Active Users):** How many users use the system daily? (e.g., 100 Million).
- **Read/Write Ratio:** Is the system read-heavy (Twitter, 100:1) or write-heavy (IoT logging, 1:100)?
- **QPS (Queries Per Second):** 
  - *Formula:* `(DAU * average_requests_per_user) / 100,000` (since there are ~86,400 seconds in a day, rounding to 100k makes math easy).

## 2. Storage & Data
- **Data Size per Request:** How big is a typical payload? (e.g., Tweet = 200 bytes, Image = 2MB).
- **Retention Period:** Do we keep data forever? (e.g., 5 years).
- **Total Storage Needed:** 
  - *Formula:* `Writes_per_second * Data_Size * Seconds_in_5_years`.

## 3. CAP Theorem Constraints
- **Consistency vs Availability:** If a network partition occurs, do we show the user an error (Consistency) or show them slightly stale data (Availability)?
- *Financial systems:* Favor Consistency.
- *Social media:* Favor Availability.

## 4. Latency Requirements
- What is the acceptable latency for a request? (< 200ms for web, < 10ms for real-time gaming).

## Example: Twitter Clarification
**Candidate:** "For Twitter, I assume we have about 300 million DAU, and each user views 100 tweets a day and posts 1 tweet. This gives us a 100:1 Read/Write ratio. Is that a fair assumption?"
**Interviewer:** "Yes, that sounds right."
**Candidate:** "Okay, so Write QPS is `(300M * 1) / 100k = 3,000 writes/sec`. Read QPS is `(300M * 100) / 100k = 300,000 reads/sec`. Because it's so read-heavy, I will definitely need to design a robust caching layer."
